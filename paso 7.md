# Paso 7 — Modelos de documentación

## 1. Modelo C4

El modelo C4 (Context, Container, Component, Code) es una técnica de diagramación jerárquica creada por Simon Brown para documentar la arquitectura de software en cuatro niveles de abstracción crecientes, permitiendo "hacer zoom" desde una vista general del sistema hasta el detalle de código. Se usa principalmente para comunicar arquitectura a audiencias distintas (negocio, arquitectos, desarrolladores) sin perder consistencia entre niveles, y encaja bien como complemento visual de un ADR o una spec técnica. Sus ventajas son la notación simple (solo cajas y flechas, sin necesidad de UML formal), la curva de aprendizaje baja y la facilidad para mantenerlo actualizado con herramientas como Structurizr o Mermaid C4. Como inconveniente, los niveles más bajos (Component/Code) tienden a desactualizarse rápido en proyectos que cambian con frecuencia, y no sustituye documentación de decisiones (para eso están los ADR) ni de contratos de API (para eso está OpenAPI). Ejemplo de referencia: el diagrama de contenedores del sistema "Internet Banking System" disponible en [c4model.com](https://c4model.com/diagrams/dynamic).

## 2. ADR (Architecture Decision Record)

Un ADR es un documento corto que registra una decisión arquitectónica significativa junto con su contexto, las alternativas consideradas y las consecuencias de adoptarla, formando un historial cronológico y trazable de "por qué" se construyó el sistema como se construyó. Se usa para evitar que el conocimiento arquitectónico viva solo en la cabeza de quien decidió, facilitando el onboarding y revisando decisiones pasadas cuando el contexto cambia. Su ventaja principal es el bajo coste de escritura (una plantilla de media página) frente al alto valor de trazabilidad histórica que aporta, además de integrarse bien en el propio repositorio como Markdown versionado junto al código. El inconveniente es que depende de la disciplina del equipo para mantenerlos actualizados y no volverlos obsoletos sin marcarlos como "superseded", y que decisiones mal delimitadas (demasiado grandes o triviales) le restan utilidad. Ejemplo de plantilla y casos reales: repositorio [joelparkerhenderson/architecture-decision-record](https://github.com/joelparkerhenderson/architecture-decision-record), con plantillas como MADR o el formato Nygard original.

## 3. Especificación OpenAPI

OpenAPI (antes Swagger) es un estándar para describir APIs REST de forma declarativa en YAML/JSON, especificando endpoints, parámetros, esquemas de request/response, autenticación y códigos de error de manera legible tanto por humanos como por máquinas. Se usa para generar documentación interactiva, clientes SDK y mocks de forma automática, sirviendo como contrato único entre equipos de backend, frontend y terceros integradores. Su ventaja principal es la interoperabilidad: al ser un estándar abierto, herramientas como Swagger UI, Postman o Redoc lo consumen sin fricción, y permite validar que la implementación real cumple el contrato documentado. Como inconveniente, los ficheros pueden crecer mucho y volverse difíciles de mantener a mano en APIs grandes, requiriendo generación automática desde el código o herramientas de modularización (`$ref`) para no perder consistencia. Ejemplo público de referencia: la especificación OpenAPI de [GitHub](https://github.com/github/rest-api-description) (github/rest-api-description) o la de [Stripe](https://github.com/stripe/openapi).
